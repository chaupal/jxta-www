General Overview

Q: What is JXTA?

A: JXTA is a research project incubated at Sun Microsystems under the guidance of Bill Joy and Mike Clary.Its goal is to explore a vision of distributed computing using peer-to-peer topology, and to develop basic building blocks and services that would enable innovative applications for peer groups. Recognizing this effort would benefit from expert coders outside of Sun (Joy’s Law: Some of the best people work elsewhere”), the project has posted a draft of a specification and some prototype code to a website, www.jxta.org under the Apache Software License, encouraging others to join in our efforts. The project now is evolving daily. 

Q: What does JXTA stand for?

A: JXTA is short for Juxtapose, as in side by side. It is a recognition that peer to peer is juxtapose to client server or Web based computing -- what is considered today’s traditional computing model. 

Q: What is prototype code?

A: JXTA is short for Juxtapose, as in side by side. Prototype is a term we have used to represent code that has been developed as proof-of-concept, to demonstrate an implementation of our ideas and specifications. It is considered usable code, with a caution that it has not been through rigorous quality assurance processes. Developers are working with it currently to understand our approach and design, and to prototype their own applications. 

Q: Who uses JXTA?

A: JXTA is short for Juxtapose, as in side by side.Today, JXTA is still mostly a research project. Developers who are interested in developing peer-to-peer applications and various individual hackers are using it to create their own prototypes. As JXTA matures, applications written based on the current design and specifications can potentially be used by a variety of users on anything from severs to PCs to pagers. 

Q: Where can I find JXTA source code? 

A: Latest builds of the source code for JXTA are posted at http://www.jxta.org. The software and code are free, subject only to the terms of the Apache Software License (modified only to reflect the JXTA name and that Sun was the original contributor). Remember though, this project is still in its research stage. The code is there for developers who are interested in understanding our approach and in contributing their ideas to the project. 

Q: How can I get involved in the JXTA?

A: We encourage contributions to JXTA. At first, we recommend you join the general mailing lists, discuss@jxta.org and participate in the forums regarding project JXTA.

 If you are a developer and wish to submit patches and suggestions to the source code, send your contributions to discuss@jxta.org mailing list. These will be reviewed by the manager in charge of that particular module, and either commented on or, more likely, implemented.

 The formal "governance" process outlining a procedure for how developers get even more involved and become what many Open Source projects call a "Committer" where you have write access to the Content Versioning System (CVS), or even a Project Owner has not been established yet. We expect the community to propose this, and to vote on a governance system by early fall. For more information about this, see our governance page. 
 
JXTA and other Technologies

Q: How is JXTA different than other P2P technologies?

A: There are several P2P technologies proposed and/or in use today, both at the application level and at the platform level. As outlined on http://www.jxta.org in our white papers and draft specifications, JXTA is a modular platform that provides simple and essential building blocks for developing a wide range of peer-to-peer services and applications. JXTA specifies a set of protocols rather than an API. Thus, JXTA can be implemented in any language on any Operating System. We anticipate solutions will range from providing a simple protocol-based wrapper that enables a small device to join a network of peers to developing a fully integrated application that supports metering, monitoring, high-level security and communication not just within a group, but also across groups on server-class systems.

The JXTA platform provides a de-centralized environment that minimizes single-points of failure and is not dependent on any centralized services. Both centralized and de-centralized services can be developed on top of the JXTA platform. JXTA services can be implemented to interoperate with other services giving rise to new distributed network computing applications. For example, a P2P communications service like instant messaging can easily be added to a resource sharing P2P application if both support the JXTA protocols. 

Q: How does JXTA fit in with Sun[tm] Open Net Environment Webtop (Sun ONE Webtop)?

A: Sun One Webtop is an open network environment whose architecture is designed to help create access, and deploy Web services on the network. This is significant to P2P because any peer on the network should have access to these Web services. JXTA enables Web services to also be distributed in a P2P environment. 

Q: What is Sun’s business model for JXTA?

A: Sun's main focus is to develop innovative systems and software that support pervasive networking. JXTA provides added value by enabling our systems and software to become active peers in a peer-to-peer network as a complement to Sun's current client/server network solutions. 

Q: How does JXTA compare to Jini[tm] technology?

A: Jini[tm] technology is a mechanism for connecting distributed services within a Java[tm] network using an object model and a centralized service location broker. JXTA is a mechanism for connecting distributed services in a peer-style network, using a loosely-coupled data model (XML). We anticipate both solutions to be used, depending on the requirements of an application. We believe Jini[tm] technology developers can implement some very powerful services to interact with JXTA distributed network computing services and applications.

Jini[tm] and JXTA show how you can get different answers to similar problems if you start from different assumptions. Jini[tm] addresses distributed computing by beginning with the assumption that Java[tm] bytecodes can be interpreted anywhere. From that base, Jini[tm] built a way of organizing networks of services and users of those services around the idea of mobile Java[tm] objects, where services advertise themselves and find other services by using the Java[tm] type system. The result is a system that is protocol independent but is tied to the Java[tm] platform, requiring at least one machine on the network to be able to run a full Java[tm] environment.

JXTA defines a set of protocols, expressed in XML, for distributed network computing. JXTA allows network entities to find other network entities, which once found can join into collaborations in application-specified ways. JXTA does not require that there be any particular platform that is supported by any of the participants of the network, but does specify the form and content of the messages that will be exchanged for network participants to find each other. 

Q: Can Jini[tm] and JXTA be used together?

A: Yes. The JXTA protocols could easily be used to find a Jini[tm] lookup service, or (more likely) to find a Jini[tm] surrogate host. Once such a surrogate host was found, the Jxta participant could load a Java object that could act as a gateway between the JXTA and Jini[tm] worlds. The details for such an approach can be found in the Jini[tm] community project for the surrogate architecture. 

Q: What is the difference between the JXTA community and the Jini[tm] community?

A: The Jini[tm] community, which has been in existence since January of 2000, is a group of developers who are doing research and development in Jini[tm] technology and products that use Jini[tm]. The Jini[tm] community requires the acceptance of the Jini[tm] Community Source License, and requires that anyone producing clients or services that are Jini[tm] compatible test those clients or services with the freely available Jini[tm] Technical Compatibility Kit. The JXTA community technology uses a slighly modified version of the Apache Open Source license, and is built around a community of researchers who are united in extending the JXTA architecture. There are no formal requirements for compatibility but compatibility ensures the largest degree of interoperability with other JXTA applications and services. 

Q: Why are there seemingly two approaches to the same problem coming from Sun?

A: Different requirements result in different systems. The Jini[tm] technology requires the existence, within a network, of at least one computer capable of running a Java environment. While this is common, it is not universal. In those cases where size or existing code require a language-neutral approach, JXTA can provide some of the basic Jini[tm] functions. 

Q: How does JXTA compare with UDDI?

A: UDDI is a centralized business registry that enables business-to-business communication. Members who register as a UDDI client become privy to a system to search and share business opportunities. It is based on XML, and for its members, provides a similar service to JXTA. 

JXTA provides a system for applications and services to announce themselves as JXTA-friendly, and create, join and monitor peer groups in a more distributed fashion. There is no sign-up process or fee; it is a more loosely organized system and will likely attract a more diverse collection of users, hackers, and businesses.

Note:  This should ultimately be expanded to include a discussion about advertisements and discovery in JXTA. 

Q: How does JXTA compare with Simple Object Access Protocol (SOAP) sponsored by the W3C?

A: Simple Object Access Protocol (SOAP), is designed to enable accessing remote services with other SOAP applications. Like SOAP, JXTA uses XML message exchange. In addition, JXTA provides protocol specifications to enable remote service access. However, JXTA provides additional protocols and services to implement a full P2P application. These include Peer Discovery to enable peers for find each other, Peer Groups to allow organization of Peers and control membership, Peer Pipes to support flexible and secure communications among peers, and Peer Monitors to allow additional management of peers in a network. It is conceivable to send SOAP compliant messages through JXTA pipes to other JXTA peers. 

Q: How does JXTA compare with Microsoft’s .NET? 

A: .NET borrows extensively from the concepts pioneered and led by SUN's Java[tm] Programming language. It consists of a common run-time execution environment and a set of requirements imposed on developer tools with the goal of enabling interoperability across the proliferation of Microsoft development languages and environments such as Visual Basic, ASP, Visual C++, and now C#. JXTA is not a programming language; it defines a set of protocols that support interoperability among Peers in a distributed network computing environment. JXTA applications can be developed in any language and can interoperate with other JXTA applications regardless of their implementation language and underlying operating system.

Technical Overview -general 

Q: What is JXTA written in? 

A: The initial prototype implementations for JXTA are written in Java programming language, as we believe the inherent qualities of Java[tm] technology add value to developing distributed network computing applications. Our specifications are intentionally language-independent however, and we support and encourage alternate implementations.  A C/C++ implementation is underway as a JXTA community effort.

Q: What are the system requirements to run JXTA? 

A: Our initial implementation for JXTA uses Java[tm] technology, and requires a platform that supports JDK[tm] 1.1.4 release or later. This includes Windows 98/ME/2000 (using IE 4.0 or later), Solaris[tm] Operating Environment, Linux, and Macintosh. We anticipate alternate implementations of JXTA will be developed to run in additional environments, including embedded operating systems, and using other development platforms such as Perl. 

Note: Some users using Windows NT[tm] or other operating environments not listed above have observed an error message, "could not find OS JRE" while trying to run the JXTA Shell. In many cases, the software can still be run on the system if a Java Run-time Environment (JRE) compatible with JDK[tm] 1.1.4 or later by running the following command from the directory where the JXTA Shell software is installed:

java -cp ../lib/jxta.jar;../lib/jxtashell.jar net.jxta.impl.peergroup.Boot

The JXTA application launchers for the reference implementation in Java[tm] have been improved by several community members to "try harder to find the JRE, but the above command may be useful for anyone wishing to know the entrypoint (starting class is net.jxta.imple.peergroup.Boot) for the JXTA reference Java[tm] implementations.

Q: What is the memory requirement to run JXTA? 

A: Our initial implementation of JXTA on JDK[tm] 1.1.4 release is around 150Kb with support for both TCP/IP and HTTP. 

Q: Does JXTA run on Java[tm] 2 Platform, Micro Edition (J2ME[tm]) and/or PersonalJava[tm] Technology? 

A: These platforms have not been tested extensively. Please keep watch on the mailing list for comments related to J2ME[tm] and PersonalJava[tm] Technology. 

Q: What are the main components of JXTA?

A: JXTA is most often described using three layers; a core layer, a middle "services" layer, and an application layer. The core layer includes the minimum protocols and building blocks to enable key mechanisms for peer to peer networking, including discovery, transport (including firewall management and limited security), and the creation of peers and peer groups. The services layer provides hooks for supporting generic services (such as searching, sharing and added security) that are used in many P2P and distributed network computing services and applications. And finally, the application layer supports implementation of integrated applications, such as music sharing, resource sharing, monetary systems, distributed storage, and other creative applications yet to be realized. These applications can be "vertical" or they can be developed to interoperate with other distributed network computing applications developed to the JXTA core. The entire system is designed to be modular, and to allow developers to pick and choose a collection of these services and applications that suits their needs. For more information on these components, review our white paper (high level overview) and/or our specifications (techies only). 

Q: What are the main JXTA protocols?

A: The JXTA platform initially defines the following protocols:

  • NetPeerGroup Protocol (Discover default NetPeerGroup) 
  • Peer Discovery Protocol (Discover new peers)
  • PeerGroup Discovery Protocol (Discover new peer groups)
  • Peer Information and Management Protocol (Peer Monitoring and Management protocol)
  • PeerGroup Membership Protocol (Join and resign from peer group)
  • PeerGroup Resolver Protocol (Resolve and Bind Core Object to a physical peer)
  • PeerGroup Sharing Protocol (Share codats in a peer group) 

Q: Who can be a Peer?

A: Peers are any devices that run some or all the JXTA protocols. A peer can be anything with a digital heartbeat that supports the JXTA core, including servers, PCs, PDAs, manufacturing and test equipment, and cellular phones. In order to participate into JXTA, the peer needs to be connected to some kind of network, such as IP, Bluetooth, or Havi. 

Q: What are Peer Groups?

A: A Peer Group is a collection of peers that have agreed upon a common set of rules to publish, share and access codats, and communicate between themselves. Each peer group can establish its own membership policy from open (anybody can join) to highly secure and protected (join only if you have sufficient credential). 

Q: What are JXTA "pipes?"

A: In JXTA, pipes provide the primary channels for communication. They are a mechanism for establishing uni-directional, asynchronous communication between peers. The pipe endpoints are non-localized to a physical peer, but are dynamically bound at creation time enabling developers to build highly available services where pipe connections can be established independently of a peer location. Pipes in JXTA are indiscriminate; they will support binary code, data strings, Java technology-based objects, and/or applets. Asynchronous pipes enable to build large-scale inter-connected services and applications.

JXTA also uses the verb "pipe" to describe concatenating commands within the JXTA shell. This is akin to the UNIX® term "piping" where standard output | standard input, etc. See discussion of JXTA Shell. 

Q: What is the JXTA Shell?

A: The JXTA Shell is an application that enables users to interact with the JXTA platform through a command-line interpreter. Much like the UNIX® shell, it is useful for accessing and managing core platform objects (peers, groups, pipes), debugging communications problems, checking the status of peers and/or peer groups, and communicating with "external" JXTA services and applications. As in UNIX®, the JXTA shell allows users to "pipe" shell commands together -- connecting commands "on-the-fly." Pipes in JXTA are useful for collecting information at one site and perhaps processing the command at another. For example, a user may query a local peer group for original music, pipe the results of query to another peer that provides a music rating service, and pipe that result to another peer providing a sorting service, next filtering the result locally to identify music of a certain genre of interest, and finally sending the result to a peer that returns concert listings for the selected music.

The JXTA Shell is not part of the JXTA core. It is an optional application designed for users and/or developers who are interested in a way to interact with the platform, monitor certain activities of their peer applications, and feed the results for another purpose. Shell commands are dynamically loaded and started by the JXTA Shell framework when they are invoked. This allows the JXTA Shell to be easily extended with new commands without having to change the source code in the shell framework and existing commands. As with the UNIX® shell, we anticipate multiple versions of a Shell application will be developed to reflect unique application requirements.

For a more complete description of the JXTA shell, refer to our white paper. 

Q: What does the term "Codat" refer to in your documentation?

A: "Codat" is a term we created to represent any computer content -- code, data, applications, or other collection of computer representable resources. JXTA does not distinguish among different types of resources that can be stored on a computer and shared among peers in a peer group.

Examples of "codat" include text files, photographs, applets, executable files, serialized Java objects, SOAP messages, etc. Codats are the elementary unit of information that is exchanged among peers. 

Q: What is InstantP2P?

A: InstantP2P is a JXTA demonstration application. It is designed to be easy to download, run, and understand. It implements the common P2P applications today of peer-to-peer messaging, peer group chat, and peer to peer file sharing. It provides a simple graphical user interface to create and join groups. In addition, the source code is readily available to provide developers a head start developing innovative new P2P and distributed network computing applications. 

Q: Where can I get the "InstantP2P" demostration application?

A: InstantP2P can be downloaded from http://download.jxta.org/easyinstall.  The source can be found at http://instantp2p.jxta.org.


Q: What dows your Content Management System (CMS) do?

A: CMS, Content Management Service, provides sharing across peers in a peer group. It implements simple searching across all peers in a peer group. CMS is used by InstantP2P and implements the search, share, and unshare JXTA Shell commands. 

Q: How does JXTA work with firewalls?

A: In order for JXTA peers to communicate with each other across a firewall, at least one peer in the peer group inside the firewall must be aware of at least one peer outside of the firewall. Furthermore, the peer inside and the peer outside the firewall that are aware of each other, must support HTTP transport, and, the firewall has to allow HTTP data transfers. These HTTP transfers need not be restricted to port 80 though that is the port used by default by most Web browsers and by the current JXTA implementation. The two peers described become routers for P2P traffic among the other peers in the peer group. There can be more than multiple peers routing P2P JXTA messages across the firewall.

Currently, the settings necessary to support firewalls and HTTP transport are made via configuration files for the JXTA Shell. We anticipate more advanced treatments of firewalls will be developed as part of basic services and higher-level applications. 

Q: Does JXTA work with NAT (Network Address Translation)? 

A: Yes. The JXTA core recognizes when a message that is being forwarded to another peer is coming from a peer with an internal (sometimes called private) IP address and automatically updates the envelope containing the JXTA message with an externally accessible IP address. 

Q: Does JXTA run using a dialup connection?

A: Yes. JXTA peers are identified by their unique ID rather than by a fixed address. When a peer boot, it attempts to contact other peers. Contacted peers contain variable sized caches that map nearby peers' unique ID to their current address. 

Q: What kind of security does JXTA use?

A: JXTA has as its goal the implementation of a model that considers what we have code-named a "P2P Web of Trust", which is in the spirit of the Pretty Good Privacy (PGP) "Web of Trust" for secure electronic mail. Just like in PGP, the "Web of Trust" is used to exchange public keys among its members. Each peer group policy may permit some members to be trusted to the extent that they have the authority to sign public keys for other members as well as to do things like authenticate, add new members, and remove or revoke membership. 

JXTA will initially implement security classes for the RSA public key exchange, the RC4 byte stream cipher, and the SHA-1 hash algorithm. These will enable privacy by the means of a P2P TLS implementation; integrity with signed hashes; non-repudiation using the "Web of Trust;" and MACs for data authenticity. Combinations of these classes form security suites, and JXTA provides the mechanism to add new customized suites as required. 

For peer group authentication a separate Pluggable Authentication Module (PAM) will be present. JXTA will have anonymous or guest login, and login with user name and password. A login session may be in clear or cipher-text as per the peer group security policy.

Since the JXTA security module is available to the core, services and applications, advanced services and applications can "plug in" their own security components and protocols. For example, the "Web of Trust" could be defined by a policy that requires authorized peer group members to be well known certificate authorities, and that peers exchange X509v3 CA signed certificates. 

Q: Can I use FTP with JXTA?

A: JXTA allows file transfers among peers over pipes using protocols other than FTP. Someone can write an FTP service for the JXTA platform that would allow also transferring files over FTP. 
 
Getting Started (Newbies) 

Q: How do I install JXTA?

A: To install the current build of JXTA, go to http://www.jxta.org and click on the "Try JXTA Now" button to download the installation program. Once the download is complete, run the installation program downloaded and follow the on-screen instructions. 

Q: Do I need to connect to the Internet before starting JXTA? 

A: No. The current JXTA implementation supports peers on a local network that need not be connected to the Internet. In addition, the JXTA specification allows peers to interact that use non-IP network protocols such as Bluetooth. However, in order for peers to interact across the Internet, at least one peer in a local peer group must have Internet access to connect to the remote peer on the Internet. 

Q: I found a bug; how do I report it?

A: JXTA is currently using Issuezilla to track bugs. Issuezilla is an open source bug tracking system developed and used by Mozilla. You use Issuezilla to browse or modify existing bugs, or enter new bugs.

To submit a bug or enhancement request for JXTA, click here.

Q: Can a JXTA peer be active in more than one JXTA peer group?

A: Yes. Joining a new peer group does not imply leaving another. However, the shell currently supports only one environment at a time. If you are using the JXTA Shell to interact with a JXTA peer group X (vs. an application that can do whatever it implements and can manage multiple groups simultaneously), and you join a new peer group Y, the shell creates a new set of environment variables for Y. The "chpgrp" command is used to change which peer group variables are active for the shell. The old services are not stopped while the shell is running so codats that were being shared still are accessible by other peers in the group. However, this does NOT imply that codats in one peer group are automatically accessible by peers in another group unless the codat is explicitly "shared" in both groups.

Q: Can you create hierarchical peer groups?

A: While peer groups are technically considered non-hierarchical, it is possible for group Y to inherit properties from group X via a pointer, but inheritance of properties are not exposed throughout JXTA. For example, the JXTA Shell recognizes one unique peer group environment at a time.

Q: Can I run multiple peer groups on a single host?

A: JXTA supports multiple peers and peer groups on a single host. This feature is controlled in the configuration file during installation. 
 
Shell FAQ
Q: Can I run non-JXTA applications from the JXTA shell?

A: Yes. For content management, MIME type information included with a codat can be used to let local applications associated with well known content types handle them automatically. However, today the shell has no specific handlers for these things.

We anticipate developers will devise "JXTA Adapters" to execute external programs.

Note:  These should be implemented with appropriate security safeguards to avoid the problems common with Microsoft office applications.

An adapter could essentially map data and connect applications for remote launches. Some examples might be: 

  • UNIX® stdio to JXTA stdio adapter - this would enable piping of JXTA commands to UNIX® commands on UNIX® platforms. This would open up a lot of possibilities on UNIX® platforms because piping is so prevalent in UNIX®
  • JXTA stream to xyz player adapter - these would be platform and application specific, but they would handle any necessary real-time data conversion between a JXTA pipe and the format required by the player. These need not be unidirectional. For example, you could adapt the output of a video capture application to become a JXTA stream.
  • HTML to JXTA studio - could be used to post and get information to/from Web pages to allow JXTA peers to interact with existing Web sites. For example, can launch a search for titles and prices on Amazon and pipe the results to other JXTA services.

Q: Can the JXTA Shell be run as an applet?

A: No. For security purposes the applet viewer used by Java restricts the network connections that can be made by an applet and the JXTA core would not be able to run inside the Applet Viewer. The Applet Viewer also prevents access to the local file system making the machine running the applet behave only as a client (aka "freeloader" in Gnutella).

Someone could write a version of the shell that runs as a signed applet to provide the functionality normally restricted by the default security of the Java Applet Viewer. A potential application for this is remote control of remote peers using a Web browser.

Q: What's the purpose of this site?

A: Jxta.org is a home site for those interested in collaborating on JXTA, including professional developers, end-users, and of course the ever valuable hacker. Here you can participate with other like-minded people who share ProjectJXTA's vision. 

As a developer and/or hacker, you can turn your ideas for JXTA core, services, and applications into active, productive projects. You can even invite developers whose expertise you seek to join you. But more than that, all the Web-based tools you need to manage your project and collaborate with other developers to get it done also reside here.

As a user, you can explore ideas related to peer-to-peer technologies. You can watch products develop, and contribute your own pet-peer-peeves before a project gets finalized. And finally, as projects mature, we expect you to be able to access JXTA services and applications here.


Q: Who else is here?

A: Other community members, including Java and C developers, open source developers, and any developers who have been invited by project leaders and other developers to collaborate on projects hosted on JXTA. 

Q: What is CollabNet's role in this site?

A:Collab.net is working with Sun Microsystems, Inc., the original creator of JXTA, to host this site as a meeting place for users and developers interested in further enhancing projectJXTA's source code, applications, and services. 

Q: Is this site only for open source projects?

A:JXTA is made available at this site under a version of the Apache Software License, altered only to reflect Sun as the original contributor, and JXTA as the referred to source code. The Apache license enables developers to create complementary services and applications either as open source products, or as proprietary products. 

This site was created to support both kinds of efforts to help people collaborate and share source code as using its collaborative tools and best practices characteristics of successful open source software development projects, while also providing those who want to stay proprietary a resource for their needs. We hope all developers consider contributing at least their fixes to bugs and general enhancements.

Q: What do you mean by "hosted tools"? 

A: Project Jxta uses a software toolset supplied by CollabNet called SourceCast to enable collaborative software development. These are Web-based tools made available to project owners and developers who become project members. SourceCast combines the best-of-breed open source tools into a Web-based software development suite. Key features of this tool set are security, scalability, extensibility, and customizability. 

Q: What are those tools?

A: SourceCast provides simple, powerful Web interfaces for version control , issue tracking , discussions and decision making , automated builds, automated testing, project management and knowledge management. Project owners can configure these tools within each project and grant roles to project participants to fit the needs of the project.

How to Participate

Q: What client-side tools will I need to participate?

A: At minimum, you need a Web browser, email client, CVS, and optionally SSH to secure your CVS connections. There might be additional domain specific tools you need depending upon the types of development project you want to contribute to. 

Q: How do I start a project?

A: Use the "Start New Project" links in the "My projects" page or subsequent pages to access the four-screen project creation process. After you submit your project information, you will be able to view and customize your new project home page. Your project will be in the "pending" status until it is approved by the site administrators. For more information, read about the process.

Q: How do I put proprietary content on my section of the site securely?

A: If your development project is proprietary, only project members can view project documents, participate in project mailing lists, download/upload project files, view and report issues, and access the CVS repository. Proprietary project owners control project membership through a member approval process and project role assignments. In addition, the project administration tool allows project owners to grant specific permissions to individual users and developers through project role assignments. 

SourceCast uses Secure Socket Layer (SSL) technology to establishes a secure connection between the user and server, with a private key to encrypt data when transferred across this connection. You can also use optional SSH tunneling to protect your CVS connections from password snooping, data corruption, or theft. For more information, read about SSH on this site. 

Q: How do I share code? 

A: When you join or lead a JXTA project, you use CVS version control to check out and work with project files, and then contribute your changes back into the project's repository. When other project members update their working files, they also have access to your code. Everyone on the CVS mailing list also receives automatic e-mail notification when files are updated. 

As a project member, you will be assigned to do code enhancements or contribute patches via IssueZilla, an issue tracking tool for managing project issues (defects, enhancements, features, tasks, and patches). You can also subscribe to project mailing lists, where you can follow and participate in ongoing discussions about the project. 

Q: How do I get other developers involved in my project? 

A: You can invite other developers to join your project at any time. When you start your own development project on jxta.org, the process includes an opportunity to list developers you want to invite to join you. They receive an automatically generated e-mail that explains a little about JXTA, gives the name and a brief description of your project, and includes the personalized message that you composed. After the project is already underway, you invite others via e-mail at your discretion. 
 
Security FAQ 

Q: What steps do you take to protect my data?

A: SourceCast take appropriate steps to protect the information you share with us from unauthorized access or disclosure. 

In addition to sharing CollabNet developer site user ID/password authentication, SourceCast implements SSL security with a digital certificate. All pages in JXTA are HTTPS secured through SSL encryption to protect any information in those pages and to authenticate that user information is sent to the correct server. Any CVS password information contained in user profiles for JXTA is also encrypted to secure project information against unauthorized access.
